# training-data-codeforces

This module is the Codeforces vertical OJ slice for `platform-training-data`.

It owns Codeforces ODS ingest, Codeforces handle-account mapping, ODS/DWD/DWM/DWS schema migrations, idempotent SQL task resources, internal Java read-side query services, public DWD/DWM/DWS query HTTP endpoints, local fixtures, and tests. Personal app-layer read queries use platform `studentIdentity` and resolve it to a Codeforces handle internally before querying warehouse repositories.

## Layer Rules

Dependency direction:

```text
web -> app -> domain
infra -> domain
config -> app/domain/infra/web
```

Rules:

- `domain` must not depend on `app`, `infra`, or `web`.
- `app` orchestrates use cases and should not build SQL.
- `infra` owns parsing, JDBC SQL, and database row mapping.
- `web` owns HTTP controllers and response DTOs only.
- `config` wires Spring beans and should not contain business logic.
- Codeforces warehouse time fields use UTC+8 (`Asia/Shanghai`) semantics.

## Directory Layout

```text
src/main/java/com/custacm/platform/trainingdata/codeforces/
  app/                 # application use cases; no HTTP or JDBC details
    service/           # ingest, DWD query, DWM query, and DWS query services
    result/            # application result records
  domain/              # domain boundary; no dependency on app/infra/web
    model/             # ODS/DWD/DWM/DWS records and read models
    value/             # reusable value objects
    criteria/          # query criteria records and default filter helpers
    parser/            # parser ports consumed by app services
    repo/              # repository/writer interfaces
  infra/               # infrastructure implementations
    parser/            # Codeforces payload parsing
    repo/              # JDBC repository/writer implementations and SQL mapping
  web/                 # HTTP layer
    controller/        # OJ-specific controllers
    request/           # HTTP request DTOs
    response/          # HTTP response DTOs
  config/              # Spring Bean wiring

src/main/resources/
  db/migration/        # Flyway migrations owned by this OJ module
  fixtures/codeforces/ # stable local Codeforces test fixtures
  sql/ods/             # ODS upsert SQL
  sql/dwd/             # DWD set-based transform SQL
  sql/dwm/             # DWM set-based transform SQL
  sql/dws/             # DWS set-based transform SQL

src/test/java/com/custacm/platform/trainingdata/codeforces/
  app/service/         # application service tests
  domain/model/        # model validation tests
  domain/criteria/     # criteria pass-through and default-helper tests
  infra/parser/        # parser fixture tests
  infra/repo/          # JDBC repository and SQL task tests
  web/controller/      # controller behavior tests
```

`target/` is generated by Maven and is not part of the maintained source layout.

| DWM personal report | `GET /api/training-data/codeforces/first-accepted/by-student` -> `CodeforcesFirstAcceptedProblemQueryService.summarizeStudentFirstAcceptedProblems` | `CodeforcesHandleFirstAcceptedProblemCriteria` | `CodeforcesFirstAcceptedProblemRepository.findHandleFirstAcceptedProblems` | `dwm_codeforces__handle_problem_first_accepted` | HTTP/app input is `studentIdentity` plus optional UTC+8 first-accepted time range and problem rating bounds; app resolves the handle, repository filters by handle, and app returns identity/handle first-accepted problem details. |
| DWM problem report | `GET /api/training-data/codeforces/first-accepted/by-problem` -> `CodeforcesFirstAcceptedProblemQueryService.summarizeProblemFirstAcceptedHandles` | `CodeforcesProblemFirstAcceptedHandleCriteria` | `CodeforcesFirstAcceptedProblemRepository.findProblemFirstAcceptedHandles` | `dwm_codeforces__handle_problem_first_accepted` | Requested problem key and optional UTC+8 first-accepted time range; app returns accepted handle count and `studentIdentity + authorHandle` list with each handle's first accepted UTC+8 time, failing if any result handle is unbound. |
| DWS app report | `GET /api/training-data/codeforces/accepted-summary` -> `CodeforcesAcceptedSummaryQueryService.summarizeStudentAcceptedProblems` | `CodeforcesAcceptedSummaryCriteria` | `CodeforcesAcceptedSummaryRepository.findDailyRatingAcceptedSummaries` | `dws_codeforces__handle_daily_rating_accepted_summary` | HTTP/app input is `studentIdentity` plus optional date/rating bounds; app resolves the handle, DB filters handle/date and can prune rows by rating columns, and app-layer aggregation returns identity/handle interval rating totals plus overall total. |
| Codeforces handle account | `CodeforcesHandleAccountService.getByStudentIdentity` | `studentIdentity` string | `CodeforcesHandleAccountRepository.findByStudentIdentity` | `codeforces_handle_account` | Public lookup returns the Codeforces handle bound to one platform `studentIdentity`. Admin create inserts one `studentIdentity + handle`; admin identity migration updates only the `student_identity` key and leaves `codeforces_handle` unchanged. |

Null time/date bounds mean no lower or upper time/date bound. Null problem rating bounds mean all rated and unrated rows. Setting either `minProblemRating` or `maxProblemRating` filters to the requested inclusive rating interval and excludes unrated rows. App-layer warehouse queries require a handle-account binding whenever they need to translate between `studentIdentity` and Codeforces handle.
